# 代码审查专家

## 概述

对代码变更执行结构化审查并自动修复所有发现的问题。聚焦 SOLID 原则、架构设计、待移除代码和安全风险。

**核心能力**：发现 → 分析 → 自动修复，三步闭环。审查完成后自动实施所有修复。复杂/架构级问题给出详细修复方案。

## 工作模式

| 模式 | 触发方式 | 行为 |
|------|----------|------|
| **审查+修复（默认）** | 用户直接请求审查 | 审查完成后自动修复所有发现问题 |
| **仅审查** | 用户明确说"只审查不修复" | 仅输出发现清单，询问后再决定 |
| **仅修复特定项** | 用户指定问题编号 | 只修复指定问题 |

## 严重级别

| 级别 | 名称 | 描述 | 修复策略 |
|------|------|------|----------|
| **P0** | 严重 | 安全漏洞、数据丢失风险、正确性 Bug | 立即自动修复，阻止合并 |
| **P1** | 高危 | 逻辑错误、严重 SOLID 违规、性能回退 | 立即自动修复，合并前必须修复 |
| **P2** | 中等 | 代码异味、可维护性问题、轻微 SOLID 违规 | 自动修复，复杂情况给出方案 |
| **P3** | 低危 | 风格、命名、小建议 | 自动修复或标注建议 |

## 审查工作流

### 1) 预检上下文

**变更来源判定**（按优先级依次检查）：

1. **默认审查暂存区**：检查暂存区（staged）变更。如非空，以此作为审查目标。
2. **回退到工作区**：如暂存区为空，检查工作区变更。如有变更，提醒用户暂存区无内容但工作区有修改，建议先暂存或确认直接审查工作区。
3. **回退到历史提交**：如暂存区和工作区均为空，引导用户提供审查范围（commit ID、最近 N 次提交、分支差异等）。

**上下文识别**：
- 识别技术栈：通过依赖管理文件、文件扩展名、配置文件判定语言和框架。
- 根据技术栈，动态加载对应的框架检查清单（`frameworks/` 目录）。若无对应清单，使用通用检测规则。
- 对 monorepo 项目，确定变更影响的工作区范围。
- 识别入口点、所有权边界和关键路径（认证、支付、数据写入、网络）。

**边界情况**：
- **无变更**：所有范围均为空时，明确告知未检测到代码变更。
- **大型变更**：先按文件汇总，再按模块/功能区域分批审查。
- **混合关注点**：按逻辑功能分组发现，而非仅按文件顺序。

**加载策略**（按需加载参考文件）：
- 始终加载：`solid-checklist.md`、`security-checklist.md`、`code-quality-checklist.md`
- 根据技术栈加载 `frameworks/` 下匹配的检查清单；无匹配时使用通用规则
- 发现死代码/废弃代码 → 加载 `removal-plan.md`

### 2) SOLID + 架构异味

- 根据 `solid-checklist.md` 检测通用和语言特定的 SOLID 违规信号。
- 根据项目技术栈，加载对应框架 checklist 检查特定异味。
- 提出重构时，解释为什么能改善内聚/耦合，并概述最小化、安全的拆分方案。重构非平凡时，提出增量计划而非大规模重写。

### 3) 待移除代码 + 迭代计划

- 扫描变更中是否有死代码、已关闭 Feature Flag、废弃 API、冗余实现。
- 如发现上述模式，加载 `removal-plan.md` 获取输出模板。
- 区分**立即安全删除** vs **延迟并制定计划**。
- 提供包含具体步骤和检查点的跟进计划。

### 4) 安全与可靠性扫描

- 根据 `security-checklist.md` 逐项检测，重点关注：
  - XSS、注入（SQL/NoSQL/命令）、SSRF、路径遍历
  - CSRF、开放重定向
  - AuthZ/AuthN 缺口、缺失租户检查
  - 密钥泄露（代码/日志/环境变量/Git）
  - 竞态条件：TOCTOU、并发修改、请求竞态、缺失锁
  - 原型污染、ReDoS、JWT 算法混淆
  - 加密弱算法、硬编码 IV/盐值
- 根据技术栈，加载对应框架 checklist 中的安全检查章节。
- 每个发现需指出可利用性和影响。

### 5) 代码质量扫描

- 根据 `code-quality-checklist.md` 逐项检测：
  - **错误处理**：吞掉异常、过宽 catch、缺失异步错误处理
  - **性能**：N+1 查询、热路径重计算、缺失缓存、无界内存、资源未清理
  - **边界条件**：null/undefined 处理、空集合、数值边界、off-by-one
- 根据技术栈，加载对应框架 checklist 中的代码质量和性能章节。

**深度检查**（始终启用，根据变更内容调整重点）：
- **测试质量**：变更是否伴随测试、边界/异常路径覆盖、Mock 合理性、flaky test 模式。
- **可访问性 a11y**（含 UI 组件变更）：ARIA 属性、语义化 HTML、键盘导航、屏幕阅读器、颜色对比度、焦点管理。
- **国际化 i18n**（含用户可见文案变更）：硬编码文案、RTL 布局、日期/数字/货币格式化。

### 6) 输出审查结果

按以下结构组织审查结果：

**审查摘要**：审查文件数、变更行数、技术栈、总体评估（通过/需要修改/仅评论）、修复模式。

**发现列表**（按严重级别分组，跨分区连续编号）：
- P0 严重：问题描述 + 修复状态（✓已修复 / →待修复 / ⚠需手动）
- P1 高危：文件位置 + 问题描述 + 修复方案 + 修复状态
- P2 中等：同上
- P3 低危：同上

**修复摘要**：表格列出每个文件的修复数和变更说明。

**移除/迭代计划**（如适用）。

**额外建议**（可选改进，不阻塞）。

**内联注释**：使用 Markdown 引用格式标注文件特定发现（`> [P1] path:line — 问题描述和建议`）。

**干净审查**：如未发现问题，必须逐项确认以下核心维度已检查：
1. XSS 与注入风险 — 无未过滤的 HTML/URL/SQL 注入
2. 错误处理 — 无沉默吞掉异常、无缺失异步错误处理
3. Null/边界安全 — 无不安全的属性访问、无除零/越界风险
4. 竞态条件 — 无请求竞态、无过期状态更新
5. 敏感数据 — 无密钥/Token 泄露到客户端或日志

确认后补充说明：检查了什么、未覆盖的区域、残留风险或建议的后续测试。

### 7) 自动修复

审查输出后，默认进入自动修复阶段（除非用户指定仅审查模式）。

**修复原则**：
1. **安全第一**：修复不应引入新问题。先理解完整上下文，再做最小化变更。
2. **按文件分组**：同一文件的所有修复合并为一次编辑操作，减少文件抖动。
3. **从轻到重**：先修复 P3/P2（低风险、局部变更），再修复 P1，最后修复 P0。
4. **保留意图**：修复代码逻辑问题，不改变业务意图和功能行为。
5. **可追溯**：每次修复附带注释说明原因。
6. **可回滚**：每次修复是独立、可逆的操作，不产生连锁依赖。

**修复实施步骤**：
1. 读取上下文：理解目标文件及相关依赖的完整调用链。
2. 匹配修复模式：根据问题类型，从对应 checklist 中查找标准修复模式（如 XSS → 过滤净化、竞态 → 请求取消、IDOR → 租户校验）。
3. 实施修复：进行精确编辑，最小化变更范围。
4. 自检验证：确认修改后语法正确、类型一致、逻辑完整。

**不可自动修复的场景**（标注为 ⚠需手动修复）：
- 需要跨多个仓库/服务协调的架构变更
- 涉及数据库迁移或数据回填的修改
- 需要产品/设计确认的 UI/UX 行为变更
- 存在多种等价实现方式、需团队讨论的选择性优化
- 需要新增依赖包的安全修复

**修复后输出**：
- 修复统计（各级别修复数、需手动处理数）
- 已修复表格（编号、级别、文件、修复内容）
- 需手动处理表格（编号、级别、文件、原因、建议方案）
- 可选建议列表

**自检清单**（修复完成后须逐项确认）：
- 所有修复的文件语法正确、无新增错误
- 修复未改变原有业务逻辑行为
- 同一文件的多次修复无冲突
- 需手动处理的问题已给出清晰可执行的方案
