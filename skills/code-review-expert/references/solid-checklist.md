# SOLID 异味检测提示

## SRP（单一职责原则）

- 文件承载不相关的关注点（如 HTTP + DB + 领域规则在同一文件）
- 大型类/模块内聚性低或有多个变更原因
- 函数编排多个不相关的步骤
- God Object 对系统了解过多
- **自问**："这个模块变更的唯一原因是什么？"

### 前端特定检测

- 页面文件同时包含：UI 渲染 + 数据请求 + 状态管理 + 工具函数
- 组件既处理展示又处理业务逻辑（应分离为容器组件和展示组件）
- Service 层混合了 API 调用、数据转换和缓存逻辑
- 单个 Hook 承担过多职责（数据获取 + 状态管理 + 副作用处理）

## OCP（开闭原则）

- 添加新行为需要编辑多个 switch/if 块
- 功能增长需要修改核心逻辑而非扩展
- 缺少插件/策略/钩子点用于变体
- **自问**："我能否在不触碰现有代码的情况下添加新变体？"

### 前端特定检测

- 添加新页面类型需要修改路由配置中的大量 if-else
- 表单验证规则硬编码在组件中，而非可配置
- 主题/样式切换需要修改多处代码
- 新增支付方式需要修改支付核心逻辑

## LSP（里氏替换原则）

- 子类检查具体类型或对基类方法抛出异常
- 重写方法弱化前置条件或强化后置条件
- 子类忽略或空实现父类行为
- **自问**："我能否在调用者不知情的情况下替换任何子类？"

### 前端特定检测

- 继承基础组件后重写生命周期但不调用 super
- 扩展基础 Service 但改变了返回数据结构
- 子组件不遵循父组件的 Props 契约

## ISP（接口隔离原则）

- 接口方法过多，大多数实现者未使用
- 调用者为狭窄需求依赖宽泛接口
- 接口方法的空/桩实现
- **自问**："所有实现者是否使用所有方法？"

### 前端特定检测

- 组件 Props 定义过于庞大，大部分属性可选且很少使用
- Context/Store 暴露过多状态，消费者只需要其中一小部分
- 工具类导出过多函数，实际只用到几个

## DIP（依赖倒置原则）

- 高层逻辑依赖具体的 IO、存储或网络类型
- 硬编码实现而非抽象或注入
- 导入链将业务逻辑耦合到基础设施
- **自问**："我能否在不改变业务逻辑的情况下替换实现？"

### 前端特定检测

- 页面直接调用网络请求 API 而非通过 Service 层
- 组件直接依赖具体的存储实现（localStorage/AsyncStorage）
- 业务逻辑直接依赖特定 UI 框架的 API

> **框架特定检测**：详见 `frameworks/react.md`、`frameworks/vue.md`

## 常见代码异味（超越 SOLID）

| 异味 | 信号 |
|------|------|
| **长方法** | 函数 > 30 行，多层嵌套 |
| **特性嫉妒** | 方法使用其他类的数据多于自身 |
| **数据泥团** | 相同参数组重复传递 |
| **基本类型偏执** | 使用字符串/数字代替领域类型 |
| **霰弹式修改** | 一个变更需要编辑多个文件 |
| **发散式变化** | 一个文件因多个不相关原因变更 |
| **死代码** | 不可达或从未调用的代码 |
| **投机性泛化** | 为假设的未来需求创建抽象 |
| **魔法数字/字符串** | 硬编码值没有命名常量 |

### 前端特定异味

| 异味 | 信号 |
|------|------|
| **状态更新滥用** | 高频调用状态更新，未使用批量/路径更新 |
| **Props 钻取** | 属性层层传递超过 3 层 |
| **巨型组件** | 单个组件文件超过 500 行 |
| **样式泄露** | 组件样式影响全局或父组件 |
| **生命周期混乱** | 在错误的生命周期中执行操作 |
| **异步竞态** | 未处理组件卸载后的异步回调 |

## 重构启发式

1. **按职责拆分，而非按大小** - 小文件也可能违反 SRP
2. **需要时才引入抽象** - 等待第二个用例
3. **保持重构增量化** - 先隔离行为再移动
4. **先保留行为** - 重构前添加测试
5. **按意图命名** - 如果命名困难，抽象可能有问题
6. **组合优于继承** - 继承创建紧耦合
7. **使非法状态不可表示** - 使用类型强制不变量

### 前端重构策略

8. **容器/展示分离** - 将数据获取与 UI 渲染分离
9. **自定义 Hook 提取** - 将可复用逻辑提取为 Hook
10. **Service 层封装** - API 调用统一通过 Service 层
11. **状态提升/下沉** - 状态放在最小必要范围
12. **视图层优化** - 高频计算移至视图层执行（如 computed/useMemo）

> **框架特定重构策略**：详见 `frameworks/` 目录下对应框架的检查清单
