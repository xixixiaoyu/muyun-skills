# 代码移除检测与计划

## 检测信号
- 无引用（grep 搜索仅返回定义位置，无调用）
- 已关闭的 Feature Flag（遥测显示 0 使用量）
- 废弃 API（含 @deprecated 标记或注释标注）
- 冗余代码（新实现已完全替代）

## 分类

**立即删除**：无消费者、纯工具函数、已确认无用、删除不产生破坏性变更

**延迟删除**：有活跃消费者、需迁移、需审批。须提供：前置条件、迁移计划、时间线、回滚方案

## 输出格式

| 位置 | 原因 | 证据 | 影响 | 操作 |
|------|------|------|------|------|
| path:line | 为什么删 | 无引用/FF关闭 | 无/低 | 1.删代码 2.删测试 3.删配置 |

## 移除前检查
- grep 搜索所有引用
- 检查动态/反射式使用
- 验证无外部消费者（API/SDK/文档）
- FF 遥测确认 0 使用量（如适用）
- 前端额外：检查埋点、A/B 测试、灰度配置是否引用

## 移除后验证
- 类型检查通过、测试通过、构建成功
- grep 确认无残留引用（函数名/组件名）
- 本地运行确认无控制台报错、功能正常
- 上线后监控：错误率无新增异常、性能指标无下降

## 自动删除执行步骤

自动修复模式下，按以下步骤执行安全删除：

### 步骤 1：确认可安全删除

使用 `grep_code` 搜索目标符号的所有引用：
```
grep_code --regex "functionName|ComponentName|variableName" --path <project_root>
```

满足以下条件才执行自动删除：
- 搜索结果仅包含定义位置（无其他调用点）
- 非公开 API / 非导出符号 / 非 SDK 入口
- 无动态引用（`eval`、`Reflect`、`obj[key]` 动态访问）

### 步骤 2：执行删除

使用 `search_replace` 删除目标代码块，同时处理：
- 删除函数/组件/变量定义
- 删除对应的 import 语句（如该 import 仅用于被删除代码）
- 删除对应的测试文件/测试用例
- 删除相关的配置项（如 Feature Flag 定义）

### 步骤 3：删除后自检

```bash
# 确认无残留引用
grep_code --regex "DeletedFunctionName" --path <project_root>

# 确认构建通过
npm run build --if-present || pnpm build --if-present

# 确认类型检查
npx tsc --noEmit 2>&1 | head -20
```

### 步骤 4：输出删除报告

```markdown
### 已删除代码
| 位置 | 类型 | 原因 | 验证结果 |
|------|------|------|----------|
| utils/deprecated.ts:15-30 | 废弃函数 | 0 引用，已替代 | grep 确认无残留 |
| config/features.ts:8 | Feature Flag | 遥测 0% 使用 | 构建通过 |
```

### 不可自动删除的场景

以下情况标记为延迟删除并输出手动计划：
- 仍有活跃消费者（需先迁移）
- 是公开 API/SDK 的一部分（需版本弃用流程）
- 涉及数据库表/列（需迁移脚本 + 回滚方案）
- 被 A/B 测试或灰度配置引用
